# Nodus P3 Lite 計畫與前置作業（2026-05-24）

## 1. 方向

這版改成輕運維、重體驗。

優先順序：
1. 使用者能快速找到內容。
2. 搜尋與學習路徑操作順。
3. 每週小步更新且不破壞核心流程。

非優先：
1. 正式值班制度與重型 incident 流程。
2. 複雜多環境 promotion gate。

## 2. 最小前置作業

### 2.1 必做底線

1. Admin key 驗證維持啟用。
2. 既有限流維持啟用。
3. 發版前固定執行 pytest 與 npm build。
4. 發版後手動 smoke：載入、搜尋、語言切換。

完成定義：
- 每次更新都能用同一組最小流程驗證可用。

### 2.2 可選項（延後）

1. 告警規則細分與優化。
2. benchmark baseline 版本化流程。
3. 集中式 log / audit 檢索。

完成定義：
- 不阻擋上線，留到流量成長後再啟用。

### 2.3 基本監控與通知（本輪採用）

1. 保留 `/api/health` 與 `/api/metrics` 可讀。
2. 將 5xx、admin trigger 異常、Neo4j 連線失敗接到單一通知通道。
3. 不建立正式值班排程；通知先由維護者在工作時段處理。

完成定義：
- 發版後若核心功能異常，維護者能被動收到通知並依 runbook 回滾。

## 3. 兩週執行（Day by Day）

### Day 1-3：首屏與導覽

1. 首屏文案改為 5 秒可理解價值。
2. 首屏加一個主 CTA：開始探索。
3. 首屏加一個次 CTA：直接搜尋。

### Day 4-6：搜尋體驗

1. 優化搜尋結果排序與空狀態提示。
2. 補快速篩選入口（領域、難度或標籤）。
3. 搜尋結果點進節點流程減少一步。

### Day 7-9：學習路徑

1. 節點側欄增加一鍵加入學習路徑。
2. 學習模式顯示進度與下一步建議。
3. 補完成回饋，讓使用者知道已學完哪些。

### Day 10-12：內容節奏

1. 建立每週內容更新批次。
2. 訂最小審稿規則（命名、摘要、標籤一致）。
3. 每次更新後跑最小驗證流程。

### Day 13-14：上線觀察

1. 用實際使用路徑檢查轉化漏斗。
2. 整理 3 個最高優先改進點。
3. 定下下週迭代清單。

## 4. 本輪已完成

1. 測試基準已更新為目前狀態（48 passed, 1 skipped）。
2. 前端 build 已驗證成功。
3. Lite 規劃已取代重運維規劃作為預設執行方案。

## 5. 驗收指標（Lite）

1. 新訪客 30 秒內完成一次有效搜尋。
2. 搜尋結果到節點點擊率提升。
3. 節點到學習路徑啟動率提升。
4. 每次更新後核心驗證維持綠燈。
